Redux 是一个流行的 JavaScript 库，用于管理应用程序的状态。

它就像一个全局对象，该对象保存您稍后在应用程序中用于各种目的的信息（例如，决定要呈现的组件以及何时渲染存储的数据等）

Redux

Redux 3 main parts

Actions、Reducers、Store

Store

存储区保存了应用程序的状态,每个应用程序仅创建一个 store。
Redux 中的 state 采用 object 的形式，通常被称为“状态树”。

Actions

Actions 是定义了什么被改变了，而不是如何改变

Redux 要求我们的操作对象包含一个 type 字段,这个字段用来说明我们正在操作哪种 action,通常应为从文件导出的常量。

另一个术语是 Action Creators.
使用它们是为了使我们可以在 action 中“插入”动态数据（换句话说，使用动态数据生成 action）

Reducers

reducer 是纯函数,用来定义这个 app 的 atate 是【如何】改变的,换句话说，它们用于重新计算新的 stare，或者至少是其中的一部分。

每当我们将 action 发送到我们的 store 时，该 action 就会传递到 reducer。
reducer 函数带有两个参数：上一个 state，正在分派的 action, 并返回新的 state。

在实际的应用程序中，Reducers 很可能会变得非常复杂。为了处理 Reducers 的复杂性，我们将它们分成多个更简单的 Reducers，然后再将它们与 Redux 帮助器函数结合使用，称作combineReducers。

主要的减速器通常称为 Root Reducer。

Data Flow（数据流）

假设用户触发了一个事件，并且程序 state 更新

  • 按钮单击处理程序函数使用store.dispatch()将 action 调度到 store
  • Redux 将调度的动作传递给减速器
  • store 保存由 reducer 返回的新状态
  • 由于我们已经订阅了 store，因此我们提供的函数将被调用，并将相应地更新 UI

start our Redux

尝试使用 redux

  • 就是用 redux 来存储 list 的对象数组，然后放到 store 里存着

  • store 可以用 redux 提供的 createStore 方法来创建,第一个参数是 reducer,第二个参数可以提供应用程序的初始状态

  • action 就是一些方法，在外面调用来处理 store 里的数据，但是其实写的时候不在函数里写逻辑，只写 type，返回 type，还有想要更新 store 所需要的参数。

  • reducer 是比较重要的，是用来修改 state 的，用来处理 action 返回的那几个参数，然后直接处理 state，return 是更新后的 state

  • 使用 redux 就是在需要的 js 文件里引入 store 和 action，然后store.dispath(action(args))，就可以调用具体的 action 方法

  • 如果是想要根据 store 更新来渲染的组件，则需要 store.subscribe，后面用箭头函数调用 render 渲染函数。

combineReducers

使用 combineReducers 函数,可以只单独处理 state 的其中一个属性，不用一次性对全部 state 进行修改。

  • 每个 state 相关的参数都有一个 reducer 文件处理，reducer 纯函数的第一个参数是这个 reducer 需要修改的那个参数

  • 添加另一个 reducer。visibilityFilter。它将按可见性处理筛选笔记。

  • state 最终变成了我们在 combineReducers 函数中提供的键的对象，并且每个 reducer 都管理着自己的 state。

React-Redux

React-Redux 提供了一个称为 Provider 的 React 组件，该组件使我们的应用程序存储在整个应用程序中可用。

React Redux 提供了一个 connect 函数。每当我们想将组件“连接”到 Redux 时，即使用 React 组件与 Redux 存储交互时，我们都会使用此函数。

  • 该函数实际上是一个高阶组件。它返回一个函数，我们向其提供组件的类名

  • 该函数带有两个可选参数：mapStateToProps 和 mapDispatchToProps。

    • mapStateToProps: 如果要 subscribe，则提供一个非空的参数, 就是把需要的 store 给 map 到现在写的组件的 prop 里,以便于在当前需要的组件调用(state, ownprops?) => stateprops。如果我们不想 subscribe，就是 null。

    • mapDispatchToProps:就是把需要的 action 给 map 到现在写的组件的 prop 里, 通常是一个对象

自己的理解

三大要素

  • store 是一个存储 state 的全局变量对象

  • action 是一个修改，传进去一个某一 state 的新值

  • reducer 是真正的修改，传入 store 和 action，返回更新的 store

建立 Redux

action 文件夹中写需要改的 action

reducer 中写 store 和 reducer 函数，可以单个写，最后在 combineReducer 里整合

store 就是在 index 文件里 import 一下,再用<Provide store={store}/>包到根目录上就可

在组件中调用和修改 store

在单文件组件中使用 connect 函数，本质上是一个高阶组件，就像是 input 上面放一个更高阶的组件，这样就可以在 input 里访问修改高阶组件里面的变量。所以 connect 里面的组件可以访问 redux 的 store。

可以访问之后，有两个方法分别是用来：① 获取 state 值（同步更新）② 修改 state 值。这就是 mapStateToProps 和 mapDispatchToProps 两个 connect 的参数

最后在组件里调用就可以了

使用 Hooks 的 Redux

理念和原理和上面的一样，就是在组件内使用 Redux 的时候，mapStateToProps 和 mapDispatchToProps 需要用useDispatch()和useSelector()来替代

思考

为什么要设计 action，为什么不能直接用 reducer 来修改 state 的状态？

假设没有 rection，reducer 传入一个 state，传出一个 new state，如何得到这个 new state 的逻辑要在这个 reducer 里实现。但是整个程序软件里，不可能只有一两个地方修改这个 state，所有的组件里对这个 state 的修改都要在一个 reducer 里实现，这个 reducer 就会变得很复杂庞大，并且无法在这个函数里合理追踪到底是哪个地方修改 state 出了问题。

所以设计成这样的函数，就保证传入的每一条 state 都有一个 action 作为追踪（根据 action 的 type 值作为追踪），这样就很容易定位到底是哪里出了问题。也会减少 reducer 里逻辑混乱造成的 bug

纯函数:相同的值总是会得到相同的输出
